## Friction, deviations, skips
- **FR-133 plugin packaging:** `@blockly/zoom-to-fit@13.1.0` + `@blockly/workspace-minimap@13.1.0` ship UMD dists; bundling them into editor-ui's ESM dist synthesized a broken `blockly/core` default-import interop that failed editor-element's IIFE build AND its Vitest suite. Fix: treat both plugins as external runtime deps exactly like `blockly` itself (external in editor-ui/react/element ESM builds; real deps of element/react so the IIFE self-bundles). **Watch-out:** the plugins' peer range is `^13.1.0` while the repo pins `blockly@13.0.0` — pnpm warns (non-fatal); consider bumping blockly to 13.1.0 at the next AD-021 pin review.
- **Real bug found only in the browser pass:** the minimap mirrors the primary workspace by replaying change events, but the mount's programmatic mutations (`loadDocument`/`clear`) run with Blockly events disabled — so the mirror stayed empty and every later replay threw `The associated block is undefined` (console error on each collapse/move/delete). Fixed with `SyncablePositionedMinimap.syncFrom()` (subclass reaches the protected `minimapWorkspace`; full-state reload after each programmatic mutation), red-first test in `navigation.test.ts`. jsdom suites had only asserted the minimap's DOM presence — a lesson for what "functional" must mean in future navigation tests.
- **Blockly implicit default:** `collapse` is already on for any `categoryToolbox` + non-read-only inject; the explicit option is contract-hardening, not a behavior change (documented in mount.ts + test).
- **Density harness split:** the plan placed the corpus sweep in `packages/editor-ui/test/`, but `encode()` needs a real engine (AD-008/AD-032) and editor-ui must not depend on the adapter (dep cycle) — the full-corpus sweep lives in `test/engine-node-adapter/test/ui/density-corpus.test.ts` (per-file jsdom override), the committed baseline stays at `packages/editor-ui/test/density-baseline.json`.

- A stale `editor-ui/dist` initially made the density numbers look like a no-op (the adapter harness resolves the built dist, and direct `npx vitest --root` bypasses turbo's `test → ^build`); caught by the implementer, rebuilt, honest before/after captured. ## Post-milestone hardening — NFR-050 geometry invariants (same day, user feedback)

User reported 5 visual geometry defects in the compact renderer (seams between stacked blocks,
off-center puzzle tabs, child protrusion, unshared left edges, near-colliding rows). Handled
SPEC-first: **NFR-050** appended (§8.5, SPEC v2.2, id-ledger 294) specifying zero-gap stacking +
shared left edge, label↔child visual alignment, and `GRID_UNIT` height quantization; then a
red-first geometry harness + fix.

- **Root causes (verified against the `blockly-v13.0.0` renderer sources):** the density pass
  shrank some constants but left `LARGE_PADDING` (spacer between external-value rows) and
  `TAB_OFFSET_FROM_TOP` at their thrasos bases — the seam reduces algebraically to
  `LARGE_PADDING − TAB_OFFSET_FROM_TOP − MEDIUM_PADDING` = exactly 2px, any child height. Fixed by
  deriving `LARGE_PADDING = TAB_OFFSET_FROM_TOP + MEDIUM_PADDING` and quantizing everything to
  `GRID_UNIT = 4`. Tab centering: the *connection coordinate* sits at the row top by thrasos
  convention (untunable; NFR-050(b) re-worded to the drawn-glyph invariant), while the visual
  alignment falls out of two pinned preconditions — `TAB_OFFSET_FROM_TOP == MEDIUM_PADDING`
  (label↔child centers cancel exactly, any height) and `2·TAB_OFFSET_FROM_TOP + TAB_HEIGHT ==
  pillHeight` (`TAB_HEIGHT` 12→16). The last odd height was the +/- mutator `FieldImage`: 16px
  icon + Blockly's private 1px `Y_PADDING` = 17px row propagating into every array/object
  ancestor — glyphs resized to `GLYPH_SIZE=15` → 16px row, zero quantization exemptions anywhere.
- **Harnesses:** `packages/editor-ui/test/geometry.test.ts` (7 tests, 5 failed red against the
  pre-fix renderer) + `test/engine-node-adapter/test/ui/geometry-corpus.test.ts` (all 121 corpus
  examples: zero-gap, shared left edge, no protrusion, quantized heights — zero exemptions; an
  unconnected socket legitimately breaks a stacked run — one initial false positive on
  `JoinEmptyWithDefaultDict` fixed in the checker, not the renderer).
- **Density trade recorded honestly:** correct geometry costs part of the phase-3 gain — corpus
  median bbox height 109→120px (pre-M6 was 127; net −5.5% instead of −14.2%), blocks-visible
  unchanged (721), `set__base` 20→24px (bound ≤28 still met). Baseline regenerated
  (`UPDATE_DENSITY=1`); NFR-050's own text: density gains never buy geometry defects.
- **Real-browser evidence:** the exact reported workspace shape (Chain → Array → [Expression op,
  Set variable, Build object] + nested Array/Get attribute/Format string/Get variable) loaded via
  an accepted §7.15 JSON edit; programmatic browser-side sweep: **zero** gaps / left-edge deltas /
  protrusions; screenshot `retro/evidence/m6-nfr050-geometry.jpeg`.

## Second hardening round — NFR-050(b) label↔connection anchoring (same day, user feedback)

User reported that on rows stretched by a tall child, the label re-centers across the stretched
height while the tab stays at the row top ("Expression op"/"Build object" floating mid-void).
This was in fact the behavior the first NFR-050(b) formulation ("label center == child center,
any height") had *specified and proven* — the feedback superseded it. NFR-050(b) amended in place
(still unmerged on PR #7, so no append-only concern): the label/field centerline anchors to the
**drawn connection tab** (`row.yPos + TAB_HEIGHT/2`) regardless of child height; short rows are
provably unchanged (anchor == stock row-center there). Constants cannot express "anchor to row
top", so the feedback explicitly sanctioned the previously-STOPped renderer-subclass override:
`CompactRenderInfo extends Blockly.thrasos.RenderInfo` overriding only `getElemCenterline_`
(field/icon on `hasExternalInput` rows). Red-first: the new stretched-row test failed against the
pre-override renderer; corpus sweep extended with per-block RenderInfo anchoring assertions over
all 121 examples. Density baseline **byte-identical** (field placement moves no row heights;
ratchet green, no regen). Browser-verified on the reported workspace: "Expression op" 10.8px from
the top of its 192px block, "Build object" 10.8px of 265px, per-row key labels at their tabs —
evidence `retro/evidence/m6-nfr050b-anchoring.jpeg`. Note the interpretation recorded: the
requested "label center == connection point y" is anchored to the drawn tab center
(`row top + TAB_HEIGHT/2`), since the literal thrasos connection coordinate is the row top and
centering a label there would push it above the block for first rows; for short rows the user's
own premise (label already coincides) matches exactly this reading.
